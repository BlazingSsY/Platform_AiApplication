# 方案二：子路径反代接入新应用（详细版）

本文档详解如何把一个**外部独立应用**通过 Nginx 网关的**子路径反向代理**接入门户平台，
使其访问地址保持为门户自身的 `http://localhost:18080/<前缀>/`，而不暴露应用自己的 IP 和端口。

> 配套文档：通用说明见 `docs/新增反代应用卡片.md`；接入原理与踩坑见 `docs/子路径反代接入应用-原理与踩坑.md`。

本文用一个虚拟例子贯穿：

> 新应用跑在**宿主机 3050 端口**，希望通过 `http://localhost:18080/myapp/` 访问。
> 接入时把所有 `myapp`、`3050` 换成你实际的前缀和端口即可。

---

## 1. 数据怎么流（先看懂这个）

```
浏览器  http://localhost:18080/myapp/...
        │
        ▼
 ① daip-gateway 网关容器（Platform_AVIC 项目）
     读 docker/nginx/default.conf，看到 /myapp/ 开头 → 转发给 3050
        │
        ▼
 ② 新应用容器（它自己的项目，监听 3050）
     自己再分发：页面 / 接口 / WebSocket
```

方案二的本质：**网关只负责把 `/myapp/` 开头的请求原样递给 3050 端口，剩下的让应用自己处理。**
所以要改两边——网关告诉它「往哪转」，应用要能「认得自己被挂在 `/myapp/` 下」。

> 门户的 React 源码（`portal-react/`）**完全不用改**。`portal-react/src/utils/openApplication.ts`
> 看到 `/myapp/` 这种「非站内路由的相对路径」会自动 `window.open` 交给网关，属设计内行为。

---

## 2. 要改的文件清单

| # | 文件 | 所在项目 | 改什么 | 改完要做什么 |
| --- | --- | --- | --- | --- |
| 1 | `docker/nginx/default.conf` | **门户平台** Platform_AVIC | 加反代 location | 重建 gateway |
| 2 | `pot_application` 表（`docker/db/init.sql` 或直接 SQL） | **门户平台** | 加卡片数据 `url=/myapp/` | 刷新页面（数据改动不用重建镜像） |
| 3 | 应用自己的 `frontend/nginx.conf` | **新应用项目** | 加 `/myapp/` 子路径分发 | 重建应用前端 |
| 4 | 应用构建参数 `base`（如 `APP_BASE`） | **新应用项目** | 以 `/myapp/` 为根构建 | 重建应用前端 |

---

## 3. 逐文件改法

### 文件 1：门户网关 `docker/nginx/default.conf`

在 `server { ... }` 内新增两段：

```nginx
# (a) 没带尾斜杠时补一个，避免相对资源路径错乱
location = /myapp {
    return 308 /myapp/;
}

# (b) 把整段 /myapp/ 透传给 3050 端口的应用容器
location ^~ /myapp/ {
    proxy_pass http://host.docker.internal:3050;   # ← 关键：不带尾斜杠！
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    # 应用有 WebSocket 才加这三行（需要文件顶部的 map，见下）
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection $connection_upgrade;
    client_max_body_size 200M;
}
```

**`proxy_pass` 尾斜杠是方案二最容易翻车的点：**

| 写法 | 浏览器请求 `/myapp/page` | 实际转发给应用 | 用在何时 |
| --- | --- | --- | --- |
| `proxy_pass http://host.docker.internal:3050;`（**不带** `/`） | → | `/myapp/page`（**保留前缀**） | 应用**以 `/myapp/` 为根构建**（推荐，见文件 4） |
| `proxy_pass http://host.docker.internal:3050/;`（**带** `/`） | → | `/page`（**剥掉前缀**） | 应用跑在根 `/` 且资源不写死绝对路径 |

> 对 React/Vue 这类 SPA，**推荐「不带尾斜杠 + 应用按子路径构建」**。SPA 打包后资源路径
> （`/assets/...`）和路由 basename 都是构建时烤死的，只有按 `/myapp/` 构建才能对齐，
> 仅靠网关剥前缀救不了，会白屏。

应用有 WebSocket 时，还要在**文件最顶部**（`server` 块外，整个文件只需一份）加：

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}
```

改完重建网关（nginx 配置打在 gateway 镜像里）：

```powershell
docker compose -f "<平台项目目录>\docker-compose.yml" up -d --build gateway
```

### 文件 2：卡片数据 `pot_application` 表

让卡片显示在首页。两种途径任选其一。

**途径 A — 应用管理界面**：登录门户 → 应用管理 → 新增一条，`url` 填 `/myapp/`。

**途径 B — SQL**：

```sql
INSERT INTO public.pot_application
(id, name, eng_name, module, icon, sequence, status, url, is_delete, version, create_date, update_date, create_user, update_user)
VALUES
(900000000000020, '我的应用', 'My App', '生产制造', 'crr', 20, 1, '/myapp/', 0, 0, now(), now(), 100000000000001, 100000000000001)
ON CONFLICT (id) DO UPDATE SET url = EXCLUDED.url, status = EXCLUDED.status, update_date = now();
```

执行（库名 `dlsc`）：

```powershell
docker compose -f "<平台项目目录>\docker-compose.yml" `
  exec -T postgres psql -U postgres -d dlsc -c "上面的 SQL"
```

字段别填错（否则卡片不显示）：

- `module` 必须是 `设计研发 / 运营管理 / 生产制造 / 算力资源` 之一；
- `status = 1`；`is_delete = 0`；`sequence` 别空。

> 建议把这条 INSERT 也同步进 `docker/db/init.sql`，以便日后用全新数据卷重建时自动带上。
> `init.sql` 只在数据卷为空时执行一次，对已有库不生效，所以已有库须靠上面的 SQL 手动加。

### 文件 3：应用自己的 `frontend/nginx.conf`（在新应用项目里）

应用容器内部也要认得 `/myapp/` 前缀，把页面、接口、WebSocket 分开处理：

```nginx
# 接口：剥掉 /myapp 前缀转给后端
location /myapp/api/ {
    proxy_pass http://backend:8080/api/;
    proxy_set_header X-Forwarded-Prefix /myapp;
    client_max_body_size 200M;
}
# WebSocket
location /myapp/ws/ {
    proxy_pass http://backend:8080/ws/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}
# 静态资源 + SPA 回退
location /myapp/ {
    alias /usr/share/nginx/html/;
    try_files $uri $uri/ /myapp/index.html;
}
```

### 文件 4：应用按子路径构建（在新应用项目里）

构建时把「根路径」设成 `/myapp/`。参数名看应用实现，常见的是 Vite 的 `base`，
或通过构建参数传入（如 `ai-Document-Review` 用的 `APP_BASE`）：

```powershell
$env:APP_BASE='/myapp/'
docker compose -f "新应用项目\docker-compose.yml" build frontend
docker compose -f "新应用项目\docker-compose.yml" up -d
```

## 4. 部署顺序

```powershell
# 1) 起新应用（让 3050 活过来）—— 在新应用项目目录
$env:APP_BASE='/myapp/'
docker compose -f "新应用项目\docker-compose.yml" up -d --build

# 2) 重建门户网关（nginx 配置打在 gateway 镜像里）—— 在 Platform_AVIC
docker compose -f "<平台项目目录>\docker-compose.yml" up -d --build gateway

# 3) 写卡片数据（若尚未写）—— 用文件 2 的 SQL
```

## 5. 验证

```powershell
Test-NetConnection localhost -Port 3050                                          # True = 应用容器在跑
(Invoke-WebRequest "http://localhost:18080/myapp/" -UseBasicParsing).StatusCode   # 200

# HTML 里资源前缀应是 /myapp/assets/...，不是 /assets/...（否则就是文件 4 没生效）
(Invoke-WebRequest "http://localhost:18080/myapp/" -UseBasicParsing).Content
```

## 6. 故障对照表

| 现象 | 根因 | 处理 |
| --- | --- | --- |
| **502 Bad Gateway** | 网关规则对了，但 3050 没起来 / 不可达 | 起应用容器；`Test-NetConnection localhost -Port 3050` |
| **404 或跳出门户首页** | 网关 location 没配或没重建 gateway | 加 location 后 `up -d --build gateway` |
| **白屏但状态 200** | 文件 4 没按 `/myapp/` 构建，资源仍是绝对根路径 `/assets/...` | 带 `APP_BASE=/myapp/` 重新构建应用前端 |
| **接口 404** | 应用容器内 `nginx.conf` 缺 `/myapp/api/` 或前缀不一致 | 对齐文件 3 的 location 与构建前缀 |
| **卡片不显示** | `module` 写错 / `status≠1` / `is_delete≠0` / `sequence` 空 / 用了旧数据卷 | 检查 `pot_application` 字段 |

---

## 7. 一句话总结

> - 卡片「显示」 → 改 **`pot_application` 表**（`docker/db/init.sql` 或运行库直接 SQL）。
> - 卡片「点开能反代访问」 → 改 **`docker/nginx/default.conf`** 并重建 `gateway`。
> - 外部反代应用，**门户 React 源码不用动**；应用那边需改 **`frontend/nginx.conf`** 并按 **子路径 base** 重新构建。
