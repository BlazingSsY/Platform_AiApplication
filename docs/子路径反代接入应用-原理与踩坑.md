# 子路径反代接入应用：原理与踩坑

本文档讲清楚一件事：把一个**独立打包的外部应用（尤其是 Vite/React、Vue 这类 SPA）**通过**子路径反向代理**挂到门户网关下时，为什么常常踩坑、根因在哪、最终应该怎么做。

适用于任意待接入应用，不针对某个具体应用。文中统一用占位写法：

- 子路径前缀：`/myapp/`（换成你的实际前缀）
- 应用容器宿主机端口：`3000`（换成你的实际端口）
- 平台项目目录：`<平台项目目录>`
- 应用项目目录：`<应用项目目录>`

> 只想要"改哪些文件"的操作清单，看 [方案二-子路径反代接入应用详解.md](方案二-子路径反代接入应用详解.md)；
> 三种接入方案的选型对比，看 [新增反代应用卡片.md](新增反代应用卡片.md)。
> 本文专注**原理与排错**，是上面两份的补充。

---

## 1. 背景与目标

门户首页「应用卡片」点击后的跳转地址来自数据库 `pot_application.url`。点击逻辑见
`portal-react/src/utils/openApplication.ts`：

- `http(s)://` 开头 → 新标签打开完整外链；
- 相对路径且**匹配站内路由** → 前端路由 `navigate`；
- 相对路径但**不是站内路由** → 新标签打开，由 Nginx 网关按路径反代。

子路径反代（方案二）的目标：让外部应用挂在门户网关下，访问地址保持为
`http://<门户地址>:18080/myapp/`，而不是暴露应用自身的 `IP:端口`。

---

## 2. 踩过的坑与根因

被接入的应用如果是 **Vite/React、Vue 等打包出来的 SPA**，独立运行在自己的容器里
（例如宿主机 `3000` → 容器内 nginx 伺服 dist），接入过程中最常见的几个坑及其根因：

| 现象 | 根因 |
| --- | --- |
| 点击后空白页（重定向报错） | 网关里 `location = /myapp { return 308 /myapp; }` **跳回自己** → 无限重定向。补斜杠应写成 `return 308 /myapp/;` |
| 重定向修好后仍空白 | 应用以 `base=/` 打包，HTML 资源是绝对根路径 `/assets/...`；经子路径访问时浏览器去**网关根路径**取资源，命中门户 SPA 返回了错误的 `index.html`（JS 被当 HTML 解析） |
| 改写资源路径后仍空白 | **关键根因**：SPA 的 **路由 basename、接口地址、WebSocket 路径都在打包时由构建参数（如 `import.meta.env.BASE_URL`）烤进产物**。`base=/` 时 basename 为 `/`，挂到子路径下路由匹配不上 → 页面根节点为空。**网关侧任何改写都救不了**（`sub_filter` 之类实测无效） |

**结论**：`base=/` 的 SPA **无法仅靠反向代理挂到子路径下**，**必须用子路径 `base` 重新构建该应用**。
所以接入这类应用前，先确认它**支持子路径构建参数**（如 `APP_BASE` / Vite `base` / `publicPath` /
`baseHref` / `context-path`）。不支持的，要么改它的构建配置，要么改用完整外部 URL（方案三）。

---

## 3. 最终方案（整段透传、不剥前缀）

```
浏览器  http://…:18080/myapp/...
   │
   ▼  daip-gateway（平台网关）
   │   location ^~ /myapp/  → proxy_pass http://host.docker.internal:3000  （不带尾斜杠，保留前缀）
   ▼  应用容器内 nginx
   │   /myapp/api/  → backend:8080/api/   （剥前缀）
   │   /myapp/ws/   → backend:8080/ws/    （剥前缀，WS 升级）
   │   /myapp/      → 静态资源 + SPA 回退 /myapp/index.html
   ▼  应用后端
```

前端以 `base=/myapp/`（例如构建参数 `APP_BASE=/myapp/`）构建后，资源前缀、路由 basename、接口、WS
全部对齐到 `/myapp/`，端到端路径一致。

> 为什么 `proxy_pass` **不带尾斜杠**：带尾斜杠会让 nginx 剥掉 `/myapp/` 前缀，而本方案要求**保留前缀**
> 透传给应用容器，由应用容器内 nginx 自行分发。尾斜杠的两种行为对照见
> [方案二-子路径反代接入应用详解.md](方案二-子路径反代接入应用详解.md)。

## 4. 改动清单

### 4.1 平台项目 `<平台项目目录>`

**① `docker/nginx/default.conf`** — 文件顶部新增 WebSocket 升级映射（应用有 WS 时才需要，整文件一份）：

```nginx
# WebSocket 连接升级映射（供 /myapp/ws 等反代使用）
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}
```

`server { … }` 内新增 / 调整 location：

```nginx
location = /myapp {
    return 308 /myapp/;
}

# 整段 /myapp/ 透传到独立应用容器（宿主机 3000）。
# 注意 proxy_pass 不带尾部斜杠/路径 —— 保留 /myapp/ 前缀不剥离。该应用需以
# base=/myapp/ 构建，资源/路由/接口/WS 全在此前缀下，由容器内 nginx 自行分发。
location ^~ /myapp/ {
    proxy_pass http://host.docker.internal:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    # WebSocket（应用 /myapp/ws/）
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection $connection_upgrade;
    client_max_body_size 200M;
}
```

> 要点：`proxy_pass` **不带尾部斜杠**（`…:3000` 而非 `…:3000/`），nginx 不剥 location 前缀，把完整
> `/myapp/...` 透传给容器。

**② `docker/db/init.sql`** — `pot_application` 应用卡片数据：

```
url = /myapp/
```

> `init.sql` 仅在 postgres 数据卷为空时执行一次；已有库需手动 UPDATE / INSERT（见 §5）。

### 4.2 应用项目 `<应用项目目录>`

**③ `frontend/nginx.conf`**（或应用容器内的 nginx 配置）— 子路径前缀对齐为 `/myapp/`，含三处 location
（接口 / WS / 静态资源回退）及 `X-Forwarded-Prefix`：

```nginx
location /myapp/api/ {
    proxy_pass http://backend:8080/api/;
    proxy_set_header X-Forwarded-Prefix /myapp;
    client_max_body_size 200M;
    # … 其余 header 略
}
location /myapp/ws/ {
    proxy_pass http://backend:8080/ws/;
    # … WS 升级 header 略
}
location /myapp/ {
    alias /usr/share/nginx/html/;
    try_files $uri $uri/ /myapp/index.html;
}
```

**④ 前端镜像构建参数**：以子路径 `base` 构建（例如 `APP_BASE=/myapp/`，对应 `vite.config.ts` 的 `base`
与 Dockerfile 的 `ARG`）。

> 原则：只改**部署配置与构建参数**，尽量不改业务源码。子路径前缀名必须四处一致——平台网关 location、
> 应用容器 `nginx.conf`、构建参数、数据库 `pot_application.url`。

## 5. 部署 / 重建步骤

> 示例：库名 `dlsc`、用户 `postgres`；网关端口 `18080`；应用容器端口 `3000`。把 `myapp`、`3000`、
> 应用卡片 `id` 换成实际值。

```powershell
# ① 更新运行中数据库（init.sql 只对空数据卷生效，已有库需手动改）
docker compose -f "<平台项目目录>\docker-compose.yml" `
  exec -T postgres psql -U postgres -d dlsc `
  -c "UPDATE public.pot_application SET url='/myapp/', update_date=now() WHERE id=<应用卡片ID>;"

# ② 以子路径 base 重建并重启应用前端容器（在应用项目目录）
$env:APP_BASE='/myapp/'
docker compose -f "<应用项目目录>\docker-compose.yml" build frontend
docker compose -f "<应用项目目录>\docker-compose.yml" up -d frontend

# ③ 重建并重启平台网关（nginx 配置打在 gateway 镜像里）
docker compose -f "<平台项目目录>\docker-compose.yml" up -d --build gateway
```

完成后**刷新门户首页**（应用列表在页面加载时拉取），点对应卡片即可。

## 6. 验证

```powershell
# HTML 资源前缀应为 /myapp/assets/...（而不是 /assets/...）
(Invoke-WebRequest "http://localhost:18080/myapp/" -UseBasicParsing).Content

# 接口经子路径反代应 200（按应用实际健康检查路径替换）
Invoke-WebRequest "http://localhost:18080/myapp/api/health" -UseBasicParsing
```

无头浏览器实测渲染（页面根节点有内容即成功）：

```powershell
$edge="C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe"
& $edge --headless=new --disable-gpu --virtual-time-budget=6000 --dump-dom "http://localhost:18080/myapp/"
```

判定标准：HTML 资源前缀为 `/myapp/assets/...` ✓；`/myapp/api/health` 返回 200 ✓；页面根节点
（如 `#root` / `#app`）内容非空 ✓。

## 7. 常见问题

- **点击后白屏**：确认应用前端是否以子路径 `base`（如 `APP_BASE=/myapp/`）构建（HTML 里资源应带该前缀）；
  若仍是 `/assets/...` 绝对根路径，说明构建参数没生效。
- **502**：网关能匹配路径但上游不可达。确认应用容器在跑、宿主机端口可达
  （`Test-NetConnection localhost -Port 3000`）。
- **接口 404 / 跨前缀**：确认应用容器内 `nginx.conf` 的 `/myapp/api/`、`/myapp/ws/` location 存在且与构建前缀一致。
- **favicon 404**：`index.html` 里 `<link rel="icon" href="/...">` 若是绝对根路径，会有一个无害 404，不影响功能。
- **改前缀名**：前缀字段必须四处一致——平台网关 location、应用容器 `nginx.conf`、构建参数、数据库
  `pot_application.url`。改完按 §5 重建网关与应用前端。
