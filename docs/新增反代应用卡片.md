# 在门户首页新增反代应用卡片

本文档说明如何在当前门户首页新增一个应用卡片，并通过 Nginx 网关反代到外部应用。

## 当前加载机制

门户首页、顶部菜单和侧边菜单使用同一套应用数据。前端会请求：

```text
GET /portal/v1/applications
```

后端从 `pot_application` 表读取应用，并按模块和 `sequence` 排序后返回。

关键字段如下：

| 字段 | 说明 |
| --- | --- |
| `name` | 应用中文名称 |
| `eng_name` | 应用英文名称 |
| `module` | 所属模块，例如 `设计研发`、`运营管理`、`生产制造`、`算力资源` |
| `icon` | 图标标识，前端按现有资源解析 |
| `status` | 状态，`1` 表示已上线可进入 |
| `url` | 点击后的入口地址 |
| `sequence` | 排序值，越小越靠前 |
| `is_delete` | 逻辑删除标记，`0` 表示正常 |

当前后端按以下模块顺序返回：

```text
设计研发 -> 运营管理 -> 生产制造 -> 算力资源
```

如果 `module` 写错，应用可能不会出现在首页分组中。

## 点击跳转规则

前端统一跳转逻辑在：

```text
portal/src/utils/openApplication.ts
```

规则如下：

- `url` 以 `http://` 或 `https://` 开头：新标签页打开完整外部地址。
- `url` 是已注册的 Vue 站内路由：检查登录状态，通过前端路由进入。
- `url` 是相对路径，但不是站内 Vue 路由：新标签页打开，由当前网关按路径反代。

因此：

```text
/sourceCodeReviewHome
```

这类路径适合站内 Vue 页面。

```text
/myapp/
```

这类路径适合挂在当前 Nginx 网关下的外部应用。

```text
http://192.168.1.20:3000/
```

这类路径适合不经过当前网关、直接跳到外部系统。

## 推荐方案选择

### 方案一：站内 Vue 页面

适用场景：

- 新页面属于当前 `portal` 前端项目。
- 路由已经在前端 `router` 中注册。
- 页面接口也可以经当前网关调用。

配置方式：

- 不需要改 `docker/nginx/default.conf`。
- 在 `pot_application.url` 写站内路由，例如 `/sourceCodeReviewHome`。
- 修改前端代码后执行：

```powershell
docker compose up -d --build gateway
```

### 方案二：同域名子路径反代

适用场景：

- 目标应用可以支持 `/myapp/` 这类子路径部署。
- 希望浏览器地址保持为当前平台的 IP 和端口，例如 `http://localhost:18080/myapp/`。

配置方式：

- 在 `docker/nginx/default.conf` 增加 `/myapp/` 的反代。
- 在 `pot_application.url` 写 `/myapp/`。
- 重建 `gateway`。

这是集成外部应用时最常用的方式。

> 详细分步操作（每个文件怎么改、`proxy_pass` 尾斜杠对照、部署顺序与故障排查）见 [方案二-子路径反代接入应用详解.md](方案二-子路径反代接入应用详解.md)；接入原理与踩坑见 [子路径反代接入应用-原理与踩坑.md](子路径反代接入应用-原理与踩坑.md)。

### 方案三：完整外部 URL

适用场景：

- 目标应用不支持 `/xxx/` 子路径。
- 目标应用已经有独立域名、独立端口或独立登录体系。

配置方式：

- 不需要改当前 Nginx。
- 在 `pot_application.url` 写完整地址，例如 `http://192.168.1.20:3000/`。

缺点是浏览器地址栏会显示外部应用自己的 IP、域名或端口。

## 子路径反代配置示例

假设外部应用运行在 Windows 宿主机的 `3000` 端口，希望通过 `http://localhost:18080/myapp/` 访问。

修改：

```text
docker/nginx/default.conf
```

增加：

```nginx
location = /myapp {
    return 308 /myapp/;
}

location ^~ /myapp/ {
    proxy_pass http://host.docker.internal:3000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

重建网关：

```powershell
docker compose up -d --build gateway
```

访问验证：

```text
http://localhost:18080/myapp/
```

## `proxy_pass` 尾部斜杠说明

Nginx 的 `proxy_pass` 是否带尾部 `/` 会影响转发路径。

如果目标应用运行在根路径 `/`，希望网关把 `/myapp/` 前缀去掉，使用：

```nginx
location ^~ /myapp/ {
    proxy_pass http://host.docker.internal:3000/;
}
```

浏览器请求：

```text
/myapp/page
```

实际转发给目标应用：

```text
/page
```

如果目标应用自己就运行在 `/myapp/`，希望保留路径前缀，使用：

```nginx
location ^~ /myapp/ {
    proxy_pass http://host.docker.internal:3000;
}
```

浏览器请求：

```text
/myapp/page
```

实际转发给目标应用：

```text
/myapp/page
```

## 新增应用卡片数据

可以通过系统里的应用管理功能维护，也可以直接执行 SQL。

示例 SQL：

```sql
INSERT INTO public.pot_application
(id, name, eng_name, module, icon, hover_icon, image, description, sequence, status, url, comments, is_delete, version, create_date, update_date, create_user, update_user)
VALUES
(900000000000010, '我的应用', 'My App', '设计研发', 'crr', NULL, NULL, '通过网关反代访问的外部应用', 10, 1, '/myapp/', NULL, 0, 0, now(), now(), 100000000000001, 100000000000001)
ON CONFLICT (id) DO UPDATE SET
    name = EXCLUDED.name,
    eng_name = EXCLUDED.eng_name,
    module = EXCLUDED.module,
    icon = EXCLUDED.icon,
    description = EXCLUDED.description,
    sequence = EXCLUDED.sequence,
    status = EXCLUDED.status,
    url = EXCLUDED.url,
    update_date = now();
```

如果 SQL 保存为 `my-app.sql`，可以在项目根目录执行：

```powershell
Get-Content .\my-app.sql -Encoding UTF8 -Raw | docker compose exec -T postgres psql -U postgres -d dlsc
```

如果只改 `pot_application` 数据，不需要重新构建镜像。刷新页面、重新登录，或清理浏览器缓存后再看首页即可。

## 验证应用是否加载

确认后端返回了应用数据：

```powershell
docker compose exec postgres psql -U postgres -d dlsc -c "select id,name,module,status,url,sequence,is_delete from pot_application order by module,sequence;"
```

确认网关可以访问门户应用接口：

```text
http://localhost:18080/portal/v1/applications
```

该接口通常需要登录态。如果浏览器未登录，可能返回认证相关信息；登录后首页应正常加载应用卡片。

## 常见问题

### 新增卡片不显示

优先检查：

- `module` 是否写成 `设计研发`、`运营管理`、`生产制造`、`算力资源` 之一。
- `status` 是否为 `1`。
- `is_delete` 是否为 `0`。
- `sequence` 是否为空。
- 当前 PostgreSQL 是否使用了旧数据卷。

### 点击卡片后打开 404

如果 `url` 是 `/myapp/`，确认 `docker/nginx/default.conf` 已经配置对应 location，并且已执行：

```powershell
docker compose up -d --build gateway
```

如果目标应用不支持 `/myapp/` 子路径，可能需要改目标应用的前端 `base/publicPath`，或者改用完整 URL。

### 点击卡片后 502

说明 Nginx 能匹配路径，但上游服务不可达。检查：

```powershell
docker compose logs gateway --tail 100
```

如果目标服务在 Windows 宿主机，确认端口可达：

```powershell
Test-NetConnection localhost -Port 3000
```

如果目标服务也在 Docker Compose 中，建议使用 Compose 服务名作为 upstream，例如：

```nginx
proxy_pass http://myapp:3000/;
```

### 修改卡片后是否需要重新构建镜像

不需要。`pot_application` 是数据库数据，修改后刷新页面即可。

只有修改以下内容时才需要构建：

- 修改 `docker/nginx/default.conf`：重建 `gateway`。
- 修改 `portal` 前端源码：重建 `gateway`。
- 修改 Java 后端源码：重建对应后端服务。
