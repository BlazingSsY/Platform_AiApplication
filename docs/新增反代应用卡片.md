# 在门户首页新增「反代外部应用」卡片

本文说明：如何在门户首页新增一张卡片，点击后跳转到经 Nginx 反向代理的**其他地址的外部应用**。

提供两种做法：
- **做法 A**：url 写完整地址（最省事）。
- **做法 B**：url 写相对路径，不写死域名（**推荐**；前端已内建公共函数，无需改代码）。

---

## 〇、先理解机制

1. **每张卡片 = 数据库表 `pot_application` 的一行**。首页接口 `/portal/v1/applications` 把该表按 `module` 字段分组返回，前端渲染成卡片。
2. **点击行为由 `url` 字段决定**（公共函数 `openApplication`）：
   - `url` 以 `http://` / `https://` 开头 → `window.open` **新标签打开**；
   - `url` 是相对路径，但不在已注册站内 Vue 路由中（如 `/myapp/`）→ `window.open` **新标签打开**，浏览器按当前平台域名解析，再经 Nginx 反代；
   - `url` 是已注册站内 Vue 路由 → 已登录时 `router.push`，未登录时弹出登录框。
3. 这段点击逻辑已统一收敛到公共函数 **`portal/src/utils/openApplication.ts`**，由三个入口组件共用：
   `FeatureCards.vue`（首页卡片）、`Header.vue`（顶部菜单）、`MenuColumn.vue`（侧边菜单）。
   该函数**已内置「反代相对路径」支持**，所以做法 B 无需再改前端。

> ⚠️ 关键：反代的外部应用在**同域名下的相对路径**（如 `/myapp/`）。旧逻辑若直接 `router.push('/myapp/')`，会被当成站内路由、匹配不到 → 页面空白。现在公共函数已内置判断：相对路径但不是站内路由时，会自动新标签打开。

`pot_application` 字段硬性约束（来自后端 `ApplicationServiceImpl.getApplicationList`）：
- `module` 必须是 `设计研发` / `运营管理` / `生产制造` / `算力资源` 之一，否则卡片不显示；
- `sequence` 不能为 NULL（否则后端排序时报错）；
- `status` = 1 卡片才可点击（0 = 未上线，灰显）；
- `icon` 不能为空：不含 `.png` 时按前端本地图标键解析（可用键见 `seed-applications.sql`）。

---

## 一、公共步骤（两种做法都要做）

### 步骤 1：加 Nginx 反代

编辑 `docker/nginx/default.conf`，仿照已有的 `^~ /circuitreview/` 加一段（把 `myapp` 和目标地址换成你的）：

```nginx
location = /myapp {
    return 308 /myapp/;
}
location ^~ /myapp/ {
    proxy_pass http://192.168.10.21:9000/;   # ← 外部应用真实地址，注意结尾的 /
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

- `proxy_pass` **结尾带 `/`**：会把 `/myapp/` 前缀剥掉再转发，适合「部署在根路径、支持子路径」的应用；
- 外部应用是否支持 `/myapp/` 子路径的判断方法，见 `docs/docker-and-app-proxy.md` 第四、五节（不支持子路径要改用子域名）。

### 步骤 2：重建网关使配置生效

Nginx 配置是打进镜像的，改完必须重建 `gateway`：

```bash
docker compose up -d --build gateway
```

### 步骤 3：新增一行应用数据

参照 `portal-backend/seed-applications.sql` 模板，往 `pot_application` 插一行。`url` 的填法在两种做法里**不同**（见下）。其它字段相同：

```sql
INSERT INTO "public"."pot_application"
  (id, name, eng_name, module, icon, status, url, sequence, is_delete, version, create_date, update_date, create_user, update_user)
VALUES
  (900000000000010, '我的应用', 'My App', '生产制造', 'crr', 1, '<URL见下>', 10, 0, 0, now(), now(), 100000000000001, 100000000000001);
```

执行方式（任选其一）：
```powershell
# 直接执行
docker compose exec -T postgres psql -U postgres -d dlsc -c "INSERT INTO ... ;"
# 或把语句写进 .sql 文件后管道导入
Get-Content my-app.sql | docker compose exec -T postgres psql -U postgres -d dlsc
```

> 也可以登录后在门户「应用管理」后台用界面新增，免写 SQL。

## 二、做法 A：不改前端代码（最省事）

**适用**：单机 / 固定部署，能接受 url 里写死平台域名。

在「公共步骤 3」里，`url` 填**完整地址**：

```
http://<平台域名>:18080/myapp/
```

例如本机就是 `http://localhost:18080/myapp/`。

点击卡片时，前端识别到 `http://` 开头 → `window.open` 新标签打开 → 请求经 Nginx 反代到外部应用。

**优点**：零前端改动。
**缺点**：url 里写死了平台域名/端口，换访问地址要同步改这行数据。

做完「公共步骤 1–3」即可，无需再动前端。

---

## 三、做法 B：相对路径、不写死域名（推荐）

**适用**：希望可移植、不想在数据里写死域名。

### 步骤 B-1：`url` 填相对路径

在「公共步骤 3」里，`url` 填：

```
/myapp/
```

### 步骤 B-2：无需改前端代码

前端已经内建公共函数 `portal/src/utils/openApplication.ts`，并由三个入口组件共用：

| 组件 | 作用 |
| --- | --- |
| `FeatureCards.vue` | 首页卡片点击 |
| `Header.vue` | 顶部菜单点击 |
| `MenuColumn.vue` | 侧边菜单点击 |

该公共函数会用已注册路由列表判断相对路径是不是站内 Vue 路由：
- `/myapp/` 这类未注册路由 → 新标签打开，经 Nginx 反代；
- `/dashboard` 这类已注册站内路由 → 走登录判断和 `router.push`；
- `http(s)://...` → 新标签打开。

因此以后新增反代应用时，**不用再改 `FeatureCards.vue` / `Header.vue` / `MenuColumn.vue`**。

### 步骤 B-3：重建 gateway

```bash
docker compose up -d --build gateway
```

**优点**：数据里只存相对路径，换部署域名无需改数据。
**缺点**：需要确认当前镜像包含公共函数版前端；如已按本文项目代码构建，则无需额外前端改动。

---

## 四、两种做法对比

| | 做法 A | 做法 B（推荐） |
| --- | --- | --- |
| 前端改动 | 无 | 无，公共函数 `openApplication` 已内建 |
| `pot_application.url` | 完整地址 `http://平台域名:18080/myapp/` | 相对路径 `/myapp/` |
| 换部署域名 | 要改数据库这行 | 不用改 |
| 适用 | 单机/固定部署、快速验证 | 可移植、长期维护 |
| 都需要 | Nginx 反代 + 重建 gateway + 加 `pot_application` 行 | 同左 |

---

## 五、验证

1. 重建 gateway 后，浏览器 `Ctrl+F5` 刷新 `http://localhost:18080/`；
2. 对应板块出现新卡片；
3. 点击卡片 → 跳转/新标签打开外部应用，且经过反代（地址栏是平台域名下的 `/myapp/`）；
4. 若卡片不显示：确认 `module` 是四个板块之一、`status=1`、`sequence` 非空；
5. 若点击空白：做法 A 确认 url 是完整 `http://...`；做法 B 确认 gateway 镜像已用当前代码重建，且 `url` 填的是 `/myapp/` 这类相对反代路径。
