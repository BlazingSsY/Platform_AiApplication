# 试验大纲审查应用（documentReview）子路径反代接入说明

本文档记录将「试验大纲审查应用」（独立项目 `ai-Document-Review`）通过**子路径反向代理**接入门户平台的完整改动、原理与运维方法。

- 平台项目：`D:\Work\9.AI\7.Aggregration_Platform\Platform_AVIC`
- 应用项目：`D:\Work\9.AI\1.Document_Review\3.Code\ai-Document-Review`
- 访问入口：`http://<门户地址>:18080/documentReview/`
- 反代路径字段：`documentReview`（早期曾用 `test-outline-review`、`ai-document-review`，现已统一为 `documentReview`）

---

## 1. 背景与目标

门户首页「应用卡片」点击后的跳转地址来自数据库 `pot_application.url`。点击逻辑见 `portal/src/utils/openApplication.ts`：

- `http(s)://` 开头 → 新标签打开完整外链；
- 相对路径且**匹配站内 Vue 路由** → 前端 `router.push`；
- 相对路径但**不是站内路由** → 新标签打开，由 Nginx 网关按路径反代。

目标：让「试验大纲审查应用」挂在门户网关下，访问地址保持为 `…:18080/documentReview/`（路径反代方式），而不是暴露应用自身的 `IP:3030`。

---

## 2. 踩过的坑与根因

该应用是 **Vite + React 打包的 SPA**，独立运行在 Docker 容器 `ai-review-frontend`（宿主机 `3030` → 容器 `80`，内部 nginx 伺服 dist）。接入过程中遇到的问题及根因：

| 现象 | 根因 |
| --- | --- |
| 点击后空白页（重定向报错） | 网关里 `location = /xxx { return 308 /xxx; }` **跳回自己** → 无限重定向 |
| 重定向修好后仍空白 | 应用以 `base=/` 打包，HTML 资源是绝对根路径 `/assets/...`；经子路径访问时浏览器去**网关根路径**取资源，命中门户 SPA 返回了错误的 `index.html`（JS 被当 HTML） |
| 改写资源路径后仍空白 | **关键**：该 SPA 的 **React Router `basename`、接口、WS 路径都在打包时由 `import.meta.env.BASE_URL` 烤进产物**。`base=/` 时 `basename='/'`，挂到子路径下路由匹配不上 → `#root` 为空。**网关侧任何改写都救不了**（`sub_filter` 方案实测 `#root` 长度仍为 0） |

**结论**：`base=/` 的 SPA 无法仅靠反向代理挂到子路径下，**必须用子路径 `base` 重新构建该应用**。所幸该应用已内置 `APP_BASE` 构建参数，支持子路径部署，属设计内的正规用法。

---

## 3. 最终方案（方案 A：整段透传、不剥前缀）

```
浏览器  http://…:18080/documentReview/...
   │
   ▼  daip-gateway（Platform_AVIC）
   │   location ^~ /documentReview/  → proxy_pass http://host.docker.internal:3030  （不带尾斜杠，保留前缀）
   ▼  ai-review-frontend 容器内 nginx（ai-Document-Review）
   │   /documentReview/api/  → backend:8080/api/   （剥前缀）
   │   /documentReview/ws/   → backend:8080/ws/    （剥前缀，WS 升级）
   │   /documentReview/      → 静态资源 + SPA 回退 /documentReview/index.html
   ▼  ai-review-backend（:8080）
```

前端以 `APP_BASE=/documentReview/` 构建后，资源前缀、路由 basename、接口、WS 全部对齐到 `/documentReview/`，端到端路径一致。

## 4. 改动清单

### 4.1 平台项目 `Platform_AVIC`

**① `docker/nginx/default.conf`** — 文件顶部新增 WebSocket 升级映射：

```nginx
# WebSocket 连接升级映射（供 /documentReview/ws 等反代使用）
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}
```

`server { … }` 内新增 / 调整 location：

```nginx
location = /documentReview {
    return 308 /documentReview/;
}

# 试验大纲审查应用：整段 /documentReview/ 透传到独立容器（ai-review-frontend，宿主机 3030）。
# 注意 proxy_pass 不带尾部斜杠/路径 —— 保留 /documentReview/ 前缀不剥离。该应用以
# APP_BASE=/documentReview/ 构建，资源/路由/接口/WS 全在此前缀下，由容器内 nginx 自行分发。
location ^~ /documentReview/ {
    proxy_pass http://host.docker.internal:3030;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    # WebSocket（应用 /documentReview/ws/）
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection $connection_upgrade;
    client_max_body_size 200M;
}
```

> 要点：`proxy_pass` **不带尾部斜杠**（`…:3030` 而非 `…:3030/`），nginx 不剥 location 前缀，把完整 `/documentReview/...` 透传给容器。

**② `docker/db/init.sql`** — `pot_application` 应用卡片数据（id `900000000000102`）：

```
url = /documentReview/
```

> `init.sql` 仅在 postgres 数据卷为空时执行一次；已有库需手动 UPDATE（见 §5）。

### 4.2 应用项目 `ai-Document-Review`

**③ `frontend/nginx.conf`** — 子路径占位前缀由 `/office-app/` 改为 `/documentReview/`，含三处 location（接口 / WS / 静态资源回退）及 `X-Forwarded-Prefix`：

```nginx
location /documentReview/api/ {
    proxy_pass http://backend:8080/api/;
    proxy_set_header X-Forwarded-Prefix /documentReview;
    client_max_body_size 200M;
    # … 其余 header 略
}
location /documentReview/ws/ {
    proxy_pass http://backend:8080/ws/;
    # … WS 升级 header 略
}
location /documentReview/ {
    alias /usr/share/nginx/html/;
    try_files $uri $uri/ /documentReview/index.html;
}
```

**④ 前端镜像构建参数**：以 `APP_BASE=/documentReview/` 构建（`vite.config.ts` 的 `base` 与 Dockerfile 的 `ARG APP_BASE` 已内置支持，未改源码）。

> 说明：仅改动**部署配置与构建参数**，未改任何 React/TS 业务源码、未改后端 Java 代码。docker 工程名 / 镜像名（`ai-document-review-frontend`）/ 容器名（`ai-review-frontend`）属工程标识，保持不变。

## 7. 常见问题

- **点击后白屏**：确认该应用前端是否以 `APP_BASE=/documentReview/` 构建（HTML 里资源应带该前缀）；若仍是 `/assets/...` 绝对根路径，说明构建参数没生效。
- **502**：网关能匹配路径但上游不可达。确认 `ai-review-frontend` 容器在跑、宿主机 `3030` 可达（`Test-NetConnection localhost -Port 3030`）。
- **接口 404 / 跨前缀**：确认应用容器内 `nginx.conf` 的 `/documentReview/api/`、`/documentReview/ws/` location 存在且与构建前缀一致。
- **favicon 404**：`index.html` 的 `<link rel="icon" href="/vite.svg">` 仍是绝对根路径，会有一个无害 404，不影响功能。
- **改前缀名**：路径字段必须三处一致——平台网关 location、应用容器 `nginx.conf`、构建参数 `APP_BASE`，外加数据库 `pot_application.url`。改完按 §5 重建网关与应用前端。
