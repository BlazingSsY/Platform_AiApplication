# 系统加载逻辑、数据库支持与启动说明

本文档按当前项目代码说明系统启动、页面加载、数据库初始化和常见维护方式。

## 当前架构

当前项目由以下部分组成：

- `portal`：Vue/Vite 前端，构建后由 Nginx 提供静态页面。
- `gateway`：Nginx 网关，统一暴露浏览器访问入口，并反代后端接口。
- `portal-backend`：门户后端，提供登录用户、组织、应用卡片、管理功能等接口。
- `circuit-review-backend`：电路审查后端。
- `sourcecode-review-backend`：软件代码审查后端。
- `logical-review-backend`：逻辑审查后端。
- `sso-server`：单点登录服务，依赖 Redis。
- `postgres`：主数据库。
- `redis`：缓存和 SSO 相关数据。
- `minio`：文件和对象存储。

外部审查引擎不包含在当前 Compose 中，需要通过 `.env` 配置地址。

## 启动流程

执行：

```powershell
docker compose up -d --build
```

整体流程如下：

1. Docker Compose 读取根目录 `.env`。
2. `postgres`、`redis`、`minio` 启动。
3. 如果 `postgres-data` 数据卷为空，PostgreSQL 自动执行 `docker/db/init.sql`。
4. 各 Java 后端服务启动，并读取 `.env` 中的数据库、MinIO、Redis、外部审查引擎配置。
5. `gateway` 启动，加载前端构建产物和 `docker/nginx/default.conf`。
6. 用户访问 `http://localhost:18080/`，浏览器加载门户页面。
7. 前端通过 `/portal/`、`/sourcecodereview/`、`/circuitreview/`、`/logicreview/`、`/sso/` 调用后端。

## 页面和应用卡片加载逻辑

门户首页应用卡片来自：

```text
GET /portal/v1/applications
```

后端读取数据库表：

```text
pot_application
```

关键规则：

- `status = 1` 表示应用已上线。
- `is_delete = 0` 表示未删除。
- `module` 决定应用分组。
- `sequence` 决定排序。
- `url` 决定点击后进入站内路由、反代路径或完整外部地址。

当前模块顺序：

```text
设计研发 -> 运营管理 -> 生产制造 -> 算力资源
```

如果首页卡片或顶部下拉为空，优先检查 `pot_application` 数据和 PostgreSQL 数据卷是否为旧版本。

## 数据库基线

当前 Docker 启动使用：

```text
docker/db/init.sql
```

该脚本是当前 Compose 使用的合并基线，包含：

- 用户、角色、权限、部门等门户基础表。
- 应用卡片表 `pot_application` 和基础应用数据。
- SSO 相关表，例如 `oauth_client`。
- 电路审查、软件代码审查、逻辑审查相关表。
- 文件、审查结果、审查明细、问题、反馈、经验分享等业务表。

正常 Docker 部署不需要手动导入各模块目录下的旧 `init.sql`。新机器使用空数据卷启动时，`docker/db/init.sql` 会自动执行。

## 数据卷和初始化行为

PostgreSQL 数据保存在 Compose 命名卷：

```text
postgres-data
```

初始化脚本只在该卷第一次创建时执行。

常见情况：

| 场景 | 行为 |
| --- | --- |
| 新机器首次启动 | 自动执行 `docker/db/init.sql` |
| 执行 `docker compose down` 后再启动 | 数据保留，不会重新导入 |
| 执行 `docker compose down -v` 后再启动 | 数据卷被删除，会重新导入 |
| 已有旧数据卷但缺表缺字段 | 不会自动修复，需要迁移或重置数据卷 |

开发环境重置数据库：

```powershell
docker compose down -v
docker compose up -d --build
```

生产环境不要直接执行 `down -v`，否则会删除已有业务数据。

## 数据库支持情况

当前 Docker 部署默认使用 PostgreSQL 14：

```yaml
postgres:
  image: postgres:14-alpine
```

当前已按 PostgreSQL 路径整理和验证。项目中部分代码或历史脚本可能保留过其他数据库适配痕迹，但当前 Docker Compose 没有提供达梦 DM8 等数据库的完整可运行配置。

如果要切换到其他数据库，需要至少处理：

- JDBC 驱动和 Maven 依赖。
- Spring 数据源配置。
- MyBatis Plus 方言和分页配置。
- SQL 类型、函数、序列、自增策略差异。
- `docker/db/init.sql` 的完整迁移。
- 所有登录、应用卡片、审查任务和管理功能回归测试。

在完成上述工作前，不建议把当前 Docker 部署直接切换到非 PostgreSQL 数据库。

## 外部审查引擎

页面和后端服务可以在当前 Compose 中启动，但执行审查任务还依赖外部审查引擎。

配置位于 `.env`：

```env
REVIEW_SERVICE_URL=host.docker.internal:38080
CODE_REVIEW_SERVICE_URL=host.docker.internal:38081
LOGIC_REVIEW_SERVICE_URL=host.docker.internal:38082
```

填写方式：

- 只写 `host:port`。
- 不要写 `http://`。
- 引擎在 Windows 宿主机时使用 `host.docker.internal:端口`。
- 引擎在局域网其他机器时使用 `IP:端口`。
- 引擎在同一个 Compose 网络时使用 `服务名:端口`。

端口不可达时，页面可能能打开，但创建或执行审查任务会失败。

## 统一网关访问

默认入口：

```text
http://localhost:18080/
```

网关路径：

| 路径 | 说明 |
| --- | --- |
| `/` | 前端页面 |
| `/portal/` | 门户后端 |
| `/circuitreview/` | 电路审查后端 |
| `/sourcecodereview/` | 软件代码审查后端 |
| `/logicreview/` | 逻辑审查后端 |
| `/sso/` | SSO 服务 |

这就是平台主页和跳转后的电路审查、代码审查、逻辑审查页面能显示同一个 IP 和端口的原因：浏览器只访问 Nginx 网关，网关在内部转发到不同服务。

## 常用维护命令

查看容器状态：

```powershell
docker compose ps
```

查看全部服务日志：

```powershell
docker compose logs --tail 100
```

查看指定服务日志：

```powershell
docker compose logs portal-backend --tail 100
docker compose logs sourcecode-review-backend --tail 100
docker compose logs gateway --tail 100
```

停止服务但保留数据：

```powershell
docker compose down
```

重新启动：

```powershell
docker compose up -d
```

修改 `.env` 后重新创建容器：

```powershell
docker compose up -d --force-recreate
```

修改代码或 Dockerfile 后重建：

```powershell
docker compose up -d --build
```

只重建网关：

```powershell
docker compose up -d --build gateway
```

## 已知问题和处理方式

### 顶部卡片、首页应用为空

常见原因是 `pot_application` 没有数据、模块名称不匹配、或 PostgreSQL 使用了旧数据卷。

开发环境可以重置数据卷：

```powershell
docker compose down -v
docker compose up -d --build
```

### 登录失败但页面不显示明确错误

先检查门户后端和网关日志：

```powershell
docker compose logs portal-backend --tail 100
docker compose logs gateway --tail 100
```

同时确认数据库中用户、角色、权限表已正确初始化。

### 左侧菜单报 `operator does not exist: bigint ~~ character varying`

这是 PostgreSQL 下把数字字段当作字符串 `LIKE` 查询造成的问题。当前代码已将部门子级查询改为等值匹配。修改后需要重建门户后端：

```powershell
docker compose up -d --build portal-backend
```

如果网关随后出现 `502`，再重新创建网关：

```powershell
docker compose up -d --force-recreate gateway
```

### 软件代码审查页面橙色空错误

优先检查 `sourcecode-review-backend` 是否正常：

```powershell
docker compose ps
docker compose logs sourcecode-review-backend --tail 100
```

如果日志中出现缺表或缺字段，说明当前 PostgreSQL 数据卷不是最新基线。开发环境建议重置数据卷；生产环境应按缺失对象补迁移 SQL。

### 镜像显示 `In use`

Docker Desktop 中镜像显示 `In use` 只表示有容器引用该镜像，包括已经停止的容器。查看运行中容器：

```powershell
docker ps
```

查看所有容器：

```powershell
docker ps -a
```

`docker compose down` 只会移除当前 Compose 项目创建的容器，不会删除镜像，也不会删除其他项目或手动创建的容器。
