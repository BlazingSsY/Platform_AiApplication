# 系统加载逻辑、数据库支持与启动说明

本文说明：①当前整体系统的加载逻辑，以及与之前的区别；②对 PostgreSQL / 达梦的支持现状；③如何启动。

---

## 一、整体加载逻辑（现状）

### 1. 组成

| 层 | 服务 | 说明 |
| --- | --- | --- |
| 入口 | `gateway`（Nginx） | 托管前端静态文件 + 反代后端各路径 |
| 后端 | `portal-backend`(8011) | 门户：登录、用户、权限、应用、资讯 |
| 后端 | `circuit-review-backend`(8012) | 电路智能审查 |
| 后端 | `sourcecode-review-backend`(8013) | 软件代码智能审查 |
| 后端 | `logical-review-backend`(8014) | 硬件逻辑智能审查 |
| 后端 | `sso-server`(9091) | 单点登录 |
| 中间件 | `postgres` / `redis` / `minio` | 数据库 / 缓存 / 对象存储 |

### 2. 启动与加载顺序

```
读取 .env
  └─ 启动 postgres / redis / minio（带 healthcheck）
       └─ postgres 首次启动：自动执行挂载的 docker/db/init.sql 完成初始化
            └─ 启动 5 个后端（depends_on: postgres/minio 健康后才起）
                 └─ 启动 gateway（依赖后端就绪）
                      └─ 浏览器访问 http://localhost:18080
```

### 3. 数据库初始化机制（关键）

- 单一合并基线文件：**`docker/db/init.sql`**（含全部表结构 + 初始数据 + 内置应用，PG 合法）。
- 它被挂载到 PostgreSQL 官方镜像的 `/docker-entrypoint-initdb.d/` 目录。
- 该目录脚本**仅在数据卷为空时自动执行一次**；已有数据则自动跳过。
- 因此：**首次启动自动建库；之后 stop/start、重启 Docker 都不会重复执行、不会清库。**

### 4. 前端加载

- 前端是构建好的静态文件，由 gateway(Nginx) 托管。
- 页面打开后，由前端发起 API（如 `/portal/v1/applications`）经 gateway 反代到对应后端。
- 应用卡片、菜单、登录等**全部依赖后端 + 数据库**；后端/数据库异常时，页面框架能显示但内容为空。

### 5. 数据持久化与配置

- 数据存放在 Docker 命名卷：`postgres-data` / `redis-data` / `minio-data`。
- 配置统一在根目录 **`.env`** 一个文件（`.env.example` 是模板）。

## 二、与之前的区别

| 方面 | 之前 | 现在 |
| --- | --- | --- |
| 数据库初始化 | 手动导入 4 个模块各自的 `init.sql`，且 `db/migration` 迁移机制被注释关闭 → schema 漂移、登录/功能反复报「缺列 / 类型不匹配」 | 合并为**单一** `docker/db/init.sql`，**容器自动挂载执行**，无需手动步骤 |
| SQL 兼容性 | 达梦经 Navicat 导出，含 PG 不兼容写法（`NULL::character varying`、`::tinyint`、外键建成 varchar、重复主键等） | 已全部修正为 PG 合法；外键统一为 `bigint`；由修好的库 `pg_dump` 生成，导入零错误 |
| 环境变量文件 | `.env.docker`，每条命令必须带 `--env-file .env.docker` | 统一为 **`.env`**，裸命令 `docker compose up -d` 即可 |
| 关闭后重启 | 初始化靠手动脚本，易丢/易重复 | **stop/start、Docker 重启数据不丢**，仅 `down -v` 会清库 |
| 镜像构建 | 各后端各自重新下载 Maven 依赖，慢且重复 | Dockerfile 启用 BuildKit 缓存挂载，依赖跨模块/多次构建共享，重建快 |
| 一键部署 | 无 | 提供 `deploy.ps1`(Windows) / `deploy.sh`(Linux)，一条命令完成全流程 |

> 原 4 个 `init.sql` 和 `db/migration` 脚本运行时**已不再使用**，仅留作历史参考。**以后改表结构，改 `docker/db/init.sql`。**

## 三、数据库支持现状

### PostgreSQL —— ✅ 已完整支持并验证

- 运行环境：Windows + Docker Desktop。
- 一导即成、可重复、重启不丢，已实测通过。
- 这是当前默认且唯一启用的数据库。

### 达梦 DM8（信创）—— 🚧 架构已设计，未落地（需 DM8 环境验证）

原则：**PG 与达梦用开关并存，不"改回来"**。需要做的改造：

| 改造点 | 当前(PG) | 达梦(DM8) | 方式 |
| --- | --- | --- | --- |
| JDBC 驱动 | `org.postgresql` | `com.dameng:DmJdbcDriver18` | pom 同时引入 |
| 数据源 | `jdbc:postgresql://`（写在 application.yml） | `jdbc:dm://` | 拆 `application-postgres.yml` / `application-dm.yml`，按 `SPRING_PROFILES_ACTIVE` 切 |
| MyBatis-Plus 分页 | POSTGRE_SQL | `DbType.DM` | 按 profile 配 `PaginationInnerInterceptor` |
| 建库脚本 | `docker/db/init.sql` | `docker/db/init-dm.sql` | 从 PG 基线做方言转换 |
| 部署 | 默认 | 指定 DM | deploy 加 `DB_TYPE` 开关 |

SQL 方言转换要点（PG→DM）：`int8→BIGINT`、`varchar→VARCHAR2`、`text→CLOB`、去掉 `COLLATE "pg_catalog"."default"` 与 `::类型` 转换、`now()→CURRENT_TIMESTAMP`、去掉 `\restrict` 等 psql 元命令、`"public".` 模式映射。

可现在就做（无需达梦）：双 profile + 驱动 + DbType 开关 + 一个 init→init-dm 转换脚本。
必须在达梦环境验证：导 `init-dm.sql` 后逐项修剩余方言差异、排查业务 SQL（MyBatis XML）里的 PG 专用写法。

## 四、如何启动

### 前置

- 安装并启动 **Docker Desktop**（托盘图标 running）。
- 首次：复制配置模板并按需修改密码/端口。

```powershell
Copy-Item .env.example .env      # Windows
# cp .env.example .env           # Linux/macOS
```

### 方式一：一键脚本（推荐）

```powershell
.\deploy.ps1 up        # Windows
./deploy.sh up         # Linux/macOS（首次先 chmod +x deploy.sh）
```

自动完成：检查 `.env` → 起基础服务 → （空库才）初始化 → BuildKit 缓存构建并启动全部。

### 方式二：原生 docker compose

```bash
docker compose up -d --build     # 首次/改代码后（含构建）
docker compose up -d             # 日常启动
```

### 访问

- 地址：`http://localhost:18080/`
- 默认账号：`admin / 111111`（管理员）；`jzadmin`、`631admin`、`631expert` 密码均为 `123456`（用 123456 登录会被要求改密）。

### 日常运维命令

```bash
docker compose ps                  # 查看状态
docker compose logs -f portal-backend   # 跟踪某服务日志
docker compose stop                # 关闭（数据保留）
docker compose start               # 重新启动（数据还在）
docker compose down                # 删除容器（卷/数据保留）
docker compose down -v             # 删除容器 + 数据卷（清库，仅测试）
```

> `deploy.ps1` / `deploy.sh` 也提供同名子命令：`up / rebuild / restart / logs / ps / down / reset / initdb`。

### 重启行为速查

| 操作 | 数据 |
| --- | --- |
| 首次启动（空卷） | 自动初始化建库 |
| `stop` → `start`、Docker Desktop 重启、开机自启 | **保留** |
| `down`（不带 -v） | 保留 |
| `down -v` / `deploy reset` | 清空（下次启动自动重建） |
